Concept

VRizzards was made in the course “Games with Motion” at TUM which focusses on games that require the player to move in a certain way and try new input mechanics. It is a VR fighting game challenging the player to summon element spells with a specific pose and fighting mystical enemies in a circular arena similar to games like Rumble. There are 3 elements: wind, earth and thunder. Additionally, the player has access to a shield to block the enemy’s attacks, activated by yet another gesture.

Despite we took some inspiration from the series “Avatar: the last air bender”, we decided on a magical fantasy setting, so we can cast magic spells on the enemies.

The game’s kickoff was in December 2023, so this page will be filled with more information later :)

We will present VRizzards at the TUM FAR DemoDay on 6th February, 2024.

Process

After we decided on the game concept, we started assigning the tasks and created a mood board for us to visualize the idea. Posture Recognition

We wanted to make the postures as easily distinguishable as possible, so we defined 3 areas based on the position of the hand controllers relative to the VR headset:

Anything above the HMD (we call it “Cam” because that’s the name of the object in Unreal), is in the first area. The area below is divided further. If the distance between the hand controller and the cam is greater than some threshold, then they are in area 3, in 2 otherwise.

As we had hardly any experience with both VR and Unreal, we printed the distances to determine the optimal threshold for defining the lowest area. For now, we hardcoded it based on the distance that felt convenient with our body heights. We plan to later add a dynamic way to compute this value during the tutorial.

To represent the areas, we defined a simple enum holding Top, Center and Bottom. We than matched each combination of left and right position to a posture. We have 3 elements in the game the player can summon: Lightning, Rock and Wind. As a start, we defined Top-Top as Lightning, Center-Center as Wind and Bottom-Bottom as Rock. You can also see the poses in our mood board:

After we managed to read out the current element, we wanted to give the player some control over when to grab an element. This way, we could avoid the player summoning a rock when accidentally dropping their hands below the threshold. So we bound the element selection to a button on the hand controllers and used the corresponding action node.

After some playing around, we wanted to be able to shoot something and see if it makes fun. We discovered the built-in Projectile blueprint and used that to shoot our elements. We found it most intuitive to shoot with the index finger button, because that is were gun triggers are placed normally. We gave it to some friends and they confirmed that it was fun and intuitive.

The next step was to refine the postures and requirements a bit more. At that point, the elements could be grabbed and shot from any hand position as long as they both were in the same area. Especially for wind (Center-Center), it was way too easy to spam projectiles. We additionally opted for having some kind of idle state for the hands, were no element could be grabbed. Like a default position, the player has to exit in order to get an element. That’s why we changed the wind posture to some thing like a T-pose. The naive approach was to simply calculate the distance of both hands on the y axis and that actually worked so well that we decided to keep it. The advantage from only comparing y (so the axis pointing to the right) is that it lets some space for inaccurate positioning of the hands. In general, we wanted to keep the requirements for any action as forgiving as possible to keep the player in the flow of the game and giving them the sense of mastering the mechanics.

Furthermore, we added the constraint that shooting a wind projectile was only possible if the hands were close enough to create a Kamehameha-like movement:

While playing, we intuitively also shot the other two elements from the center, but we didn’t restrict the ability to shoot in code as we wanted to give the player the ability to shoot lightning and rock from any position.

The next change we made was only requiring one hand to press either the summon or shoot button in order to start that action. This also let room for inaccuracies like not pressing both buttons in the same frame.

Programming and testing had already sparked a lot of great ideas for additional attacks and postures, but we wanted to first finish all the basic attacks for the elements. We felt like having a default shooting action shared by all elements gave new players less things to learn and think about. But we noted down all our ideas to be able to add them later after reaching our MVP goals. We also used this as general overview over the features and the progress we made:

Attack Cone

During testing and coding, we discovered a major challenge: aiming with two pointers, being the two hands. As it seems, it is not that obvious in which direction to shoot an element projectile if you are not aiming in one direction (as when e.g. using the HMD and look in the desired direction), but rather define a broader space to fire at between your hands. Aiming with the two hands seemed the most intuitive for us, as the final vision is to change the shooting action from a button click to some push action done by quickly moving your hands forward. Then it only makes sense to push the projectiles in the direction the player pushed in. We felt that taking the averaged direction between the hands would be the most reasonable solution to this.

Since our first project meeting, we were talking about some kind of aiming assistant for the game. The naive approach would be to just cast a ray and see which enemy was hit, but that would not account for any imprecisions in shooting. On the contrary, it would even be redundant as we solved the “hitting an enemy”-problem with collisions between the projectiles and the players.

After one of our coding sessions in university, I took the train home with Michl and he came up with the idea to have some kind of collider area in front of the player which detects all enemies in the approximate direction the player aims at. We would the steer the projectiles to the enemy that was most likely targeted by the player. The initial idea was to use a cylindric or cuboid trigger area, but we eventually decided on using a cone shape as it approximates the area the best.

I was so hooked by that plan, that I immediately started working on the cone the next day.

As I said earlier, we decided on aiming with two hands. So I tried to come up with some way to calculate where the player actually wants to shoot at. And then orient the attack cone so it captures all enemies in the area spanned by the hand vectors. I first focused on the shooting direction. As I always do when it comes to planning out how to implement something (especially if it involves math), I briefly sketch it in OneNote:

This helps me to visualize and better understand the problem. Additionally, I can detect misleading solutions much quicker. A bonus is that I am able to store my thoughts and ideas somewhere, so I won’t forget them.

Here, I started by drawing a simple 2D coordinate system with vectors pointing to the hand positions (yellow and blue). 2D is sufficient as we are only interested in the directions on the XY plane to determine the forward vector of the element projectile to spawn and shoot. I then went ahead and drew in the solution vector (red) to see if I can derive any mathematical connection to the hand vectors. I quickly realized that the middle vector lies on the diagonal of the parallelogram defined by the hand vectors. In the formula I put there, it is exactly half of the diagonal, but I later normalized it to serve only as direction indicator and not as some force, so the exact length doesn’t matter.

So the first approach was born. After that, I continued by thinking of any edge cases that the formula couldn’t compute correctly. I thought of the different constellations the hands could end up when shooting and what I would expect the direction to be in that situation. This is where the second screenshot comes into play. If one of the hands is further in front than the other, the direction would be pointing more in the direction of the nearer hand. So I needed to make the calculation independent of the distances from the player to their hands. And the go-to way to eliminate any factors in vector lengths is to normalize them. That, in fact, yielded very nice results.

Now it was time to add the cone. For this I, again, started by sketching. I then created a cone trigger object in Unreal and added it to the VR pawn:

I began with positioning the cone correctly. I thought I simply needed to make it child of the player to make it move and orient with them. But yet again, the fact that we have two hands caused some problems. I needed to position the cone in the middle of the two at any time. Some scribbling later, I found that I could just take the average of the two hand positions and set the cone’s location to the resulting vector.

I struggled with orienting the cone correctly though, it always seemed a bit off. I found out that one can change an object’s pivot point in Unreal, but as I was working in the cone blueprint itself, I didn’t manage to pull it off. I gave up on that as it was a dead end for me, and tried something different. I made the cone child of an invisible sphere acting as the pivot point. So I changed the blueprint event graph to rotate and position the sphere instead of the cone. That fixed it!

Till now I was simply printing a message to the screen log any time the cone overlapped with any enemy actor. The next step was to actually store the enemy that is most likely to be targeted. I created a variable for this that is written any time an overlap was detected. That meant that the projectiles were always targeting the last enemy that was in the cone, which wasn’t always the intuitive choice.

After some discussions with the others, we decided on storing a list of enemies that are currently in the cone. When the player hits the shoot button, I will choose the enemy that has the smallest distance to the player (being the HMD actor) as target for the homing projectiles (provided by Unreal). If the list was empty, I would simply take the middle vector and shoot in that direction without controlling the projectile’s movement further.

After I was happy with the cone, I let my friends play it and asked for their feedback. While watching them play, I discovered that they often thought they would aim at an enemy but were missing. That often happened with lightning, because they were shooting with their hands still decently high and the cone didn’t detect the enemies because they were simply too short. To mitigate that, I adjusted the cone’s vertical scale, so it would catch all enemies regardless of their height.

A future addition I will have to make is to enlarge the cone in the x axis so that it reaches from one side of the arena to the other. But I will have to wait for the level blockouts to be finished. I assume we will have a public variable for the current arena’s diameter. But for now, I’m done with the cone :)

Lightning Projectiles

Michl had basically done the core of both rock and wind projectiles. I extended those to include homing projectiles and made some other small additions. I also helped a lot with the maths.

I had just finished the attack cone, which was mainly important for the lightning attack because here, the player won’t actually shoot some projectile, but rather spawn a thunder cloud above the enemy’s head and strike him with a thunder bolt. That’s why we needed the cone in the first place. So after that was done, I went on implementing the lightning attack.

Unlike wind and rock, the event graph canvas for handling lightning was completely blank, so I had to start from scratch. I took some inspiration from what Michl has done with the other elements concerning how to spawn a projectile and set is speed and moving direction. It actually helped me a lot to write some kind of pseudo code by hand before adding any blueprint nodes. For the “miss the enemy” part I first had to think about how missing should actually be represented in this situation.
